# Phase 17: Threaded Similarity Map Plan

## Scope

Add an embedding-driven similarity surface that groups discussion messages before formal categorisation, while preserving the existing LLM-driven argument map as a separate feature.

This phase should:

- use Convex built-in vector search as the first implementation
- embed each newly submitted message asynchronously after submit/send
- keep the default display model in thread mode
- support two semantic perspectives:
  - thread mode for default display
  - atomic message mode as a secondary semantic model for future analytics
- keep app categories distinct from machine-generated semantic clusters
- keep the current D3 argument map separate and unchanged in meaning

This phase should not:

- replace app categories with semantic clusters
- merge the similarity map and argument map into one visualization
- block post submission on embedding generation
- introduce an external vector database

## Locked Decisions

The following decisions are treated as fixed inputs for implementation:

- Use Convex built-in vector search first.
- Keep `Similarity Map` and `Argument Map` as two separate surfaces.
- Default all user-facing displays to thread mode.
- Every normal discussion message except `fight_me_turn` gets an embedding.
- Categories belong to top-level thread roots, not replies.
- Replies remain nested under their root post in all default views.
- Replies may still have their own independent embeddings for semantic analysis.

## Current State

The current repo already has most of the semantic plumbing needed for phase 1:

- `semanticEmbeddings` and `semanticEmbeddingJobs` already exist in [convex/schema.ts](../convex/schema.ts).
- `semanticEmbeddings` already has a Convex vector index: `by_embedding`.
- Embeddings are currently generated by instructor-triggered batch jobs in [convex/semantic.ts](../convex/semantic.ts).
- Novelty signals are already computed from submission embedding distance in [convex/semantic.ts](../convex/semantic.ts).
- The D3 argument graph is already implemented as a separate LLM-link-driven surface in [convex/argumentMap.ts](../convex/argumentMap.ts) and [src/components/instructor/argument-map-graph.tsx](../src/components/instructor/argument-map-graph.tsx).
- Thread mode already exists in the participant workspace and treats only top-level messages as thread roots in [convex/participantWorkspace.ts](../convex/participantWorkspace.ts).

The main gaps are:

- no embed-on-submit path
- no nearest-neighbor cluster assignment path
- no semantic cluster tables
- no thread-based similarity board UI
- current categorisation logic still processes all submissions, including replies

## Convex Constraint

Convex vector search must run in an `action` via `ctx.vectorSearch`, not in a query or mutation.

That means the implementation should:

- embed and search inside actions
- persist assignment results into normal Convex tables
- expose the resulting cluster state through normal reactive queries

This is the key architectural constraint for phase 1.

## Product Model

### 1. Similarity Map

The similarity map is embedding-driven.

It answers:

- which ideas are semantically close
- which roots look duplicate or near-duplicate
- which roots are outliers
- which replies drift semantically from the root idea

Its default user-facing rendering is thread mode:

- each cluster shows root thread cards
- replies remain visually nested under the root card
- the cluster placement belongs to the root thread

### 2. Argument Map

The argument map remains LLM-link-driven.

It answers:

- which ideas support each other
- which ideas contradict each other
- which ideas extend or question each other

It should remain a separate instructor-facing surface and should not inherit vector semantics by default.

### 3. Categories vs Semantic Clusters

`categories` remain instructor-facing workflow buckets.

`semantic clusters` are machine-generated provisional groupings derived from embeddings.

They must remain separate data models:

- semantic cluster = semantic proximity
- category = instructor-approved discussion bucket

The instructor may later promote a semantic cluster into a category, but the cluster should not automatically become one.

## Semantic Rules

### Root Posts

Top-level messages are the visible clustering unit in default thread mode.

Top-level means:

- no `parentSubmissionId`
- no `followUpPromptId`
- `kind` is `initial` or `additional_point`

### Replies

Replies should be embedded independently, but default display behavior stays thread-first.

Rules:

- visible similarity placement follows the root thread
- reply embeddings are stored for future analytics
- reply-level drift may later be surfaced as metadata, but should not break thread layout in phase 1

### Fight Messages

Exclude `fight_me_turn` from similarity map embedding and clustering.

This matches the current semantic embedding batch pipeline behavior and keeps debate turns from polluting classroom similarity structure.

### Follow-Up Responses

Follow-up prompts may produce `additional_point` or reply-like behavior depending on context.

Phase 1 rule:

- if it is a top-level contribution, it may cluster as a root
- if it is structurally a reply, it remains nested under the root

## Suggested Similarity Assignment Heuristic

Phase 1 should use a simple stable heuristic instead of a complex clustering algorithm.

Suggested approach:

1. When a new root message is embedded, search nearest submission embeddings within the same question.
2. Ignore replies for the visible root-clustering pass.
3. Take top `k` neighbors.
4. If enough strong neighbors belong to one active semantic cluster, join that cluster.
5. Otherwise create a new provisional cluster.
6. Mark very low-confidence assignments as outliers.

This favors stability and understandable behavior over mathematically perfect clustering.

- The argument map remains a separate LLM-link-driven surface.

## Risks

### Cluster Instability

If re-clustering is too aggressive, the UI will feel random.

Mitigation:

- use incremental assignment for new submissions
- rebuild only when explicitly triggered

### Reply Drift

A reply may be semantically closer to a different cluster than its parent root.

Mitigation:

- keep thread mode authoritative for visible placement in phase 1
- store reply-level semantic membership for future atomic mode

### Cost Growth

Embedding every message increases AI cost and background work volume.

Mitigation:

- use the existing small embedding model
- skip fight turns
- track embedding spend in observability

### Threshold Quality

Similarity thresholds may be too strict or too loose initially.

Mitigation:

- isolate threshold constants
- add rebuild tools
- test with seeded/demo data and one real session
